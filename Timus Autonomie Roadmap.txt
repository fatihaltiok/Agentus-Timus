  Meilenstein 1 — Scheduler vollständig aktivieren

  Ziel: Timus handelt alle 15 Minuten ohne User-Input

  Was fehlt:
  - Scheduler läuft isoliert, führt aber keine echten Agent-Tasks aus
  - _check_pending_tasks() liest tasks.json — aber wer schreibt da rein?
  - on_wake Callback ist nicht mit run_agent() verbunden

  Aufgaben:
  1. Scheduler-Hook in main_dispatcher.py einbauen: on_wake → run_agent()
  2. tasks.json Schema definieren (Format für autonome Tasks)
  3. CLI-Befehl: timus task add "Recherchiere Bitcoin-Preis täglich um 9 Uhr"
  4. Scheduler beim Server-Start automatisch starten

  Tests M1 → M2:
  ✓ pytest tests/test_scheduler.py (erweitert)
  ✓ Integrationstest: Task in tasks.json schreiben → 15 min warten → wurde ausgeführt?
  ✓ Kompatibilitätstest: Läuft Scheduler parallel zu main_loop() ohne Race-Condition?
  ✓ Edge-Case: Was passiert wenn ein Task länger als 15 min dauert?

  ---
  Meilenstein 2 — Telegram-Bot als Kommunikationskanal

  Ziel: Du schreibst Timus auf Telegram → er arbeitet autonom → antwortet wenn fertig

  Was gebaut wird:
  - gateway/telegram_gateway.py — neues Modul
  - Telegram-Bot empfängt Nachricht → run_agent() → Antwort zurück
  - Langläufige Tasks: "Arbeite daran, ich melde mich..." mit Fortschritts-Updates
  - Session-Mapping: Telegram-User-ID ↔ Timus-Session-ID

  Abhängigkeit von M1: Scheduler muss stabil laufen (Tasks können via Telegram hinzugefügt werden)

  Tests M2 → M3:
  ✓ Unit: Telegram-Message-Parsing korrekt?
  ✓ Integration: Nachricht senden → Agent antwortet?
  ✓ Kompatibilität: Telegram-Session und CLI-Session gleichzeitig → kein Conflict?
  ✓ Stress: 5 Nachrichten gleichzeitig → Lane-Manager verhindert Race-Conditions?
  ✓ Edge-Case: Telegram offline → graceful degradation?

  ---
  Meilenstein 3 — Autonome Task-Queue & Priorisierung

  Ziel: Timus verwaltet eine echte Aufgabenliste, die er selbstständig abarbeitet

  Was gebaut wird:
  - orchestration/task_queue.py — persistente Queue (SQLite statt JSON)
  - Prioritäten: CRITICAL > HIGH > NORMAL > LOW
  - Task-Typen: scheduled (Cron-Zeit), triggered (Webhook), delegated (von Agent)
  - Timus kann selbst Sub-Tasks erstellen und delegieren
  - Fortschritts-Tracking im Canvas-Store

  Abhängigkeit von M2: Telegram als Kontrollkanal für Queue-Verwaltung

  Tests M3 → M4:
  ✓ pytest tests/test_task_queue.py (neu)
  ✓ Integration: Agent erstellt Sub-Task → wird ausgeführt → Ergebnis in Canvas?
  ✓ Kompatibilität: Tasks.json Migration → SQLite (Rückwärtskompatibilität)
  ✓ Edge-Case: Task schlägt fehl → bleibt in Queue oder wird dropped?
  ✓ Load-Test: 20 Tasks gleichzeitig → Priorisierung korrekt?

  ---
  Meilenstein 4 — Self-Healing & Model-Failover

  Ziel: Wenn ein Agent/Model versagt, macht Timus automatisch weiter

  Was gebaut wird:
  - Model-Failover-Tabelle: z.B. deepseek-reasoner → claude-sonnet → gpt-5-mini
  - Retry-Policy pro Task-Typ (3 Versuche, exponential backoff)
  - Fehler-Klassifizierung: API_ERROR, TIMEOUT, CONTENT_FILTER, TOOL_FAIL
  - Bei TOOL_FAIL: alternativen Tool-Pfad versuchen
  - Alert via Telegram wenn alle Retries erschöpft

  Abhängigkeit von M3: Task-Queue muss Retry-Status tracken

  Tests M4 → M5:
  ✓ pytest tests/test_model_failover.py (neu)
  ✓ Simulierter API-Ausfall → Failover-Model übernimmt?
  ✓ Kompatibilität: Failover-Antwort ist gleich formatiert wie Primär-Antwort?
  ✓ Edge-Case: Alle Models down → Telegram-Alert wird gesendet?
  ✓ Integration: Fehlgeschlagener Task landet wieder korrekt in Queue?

  ---
  Meilenstein 5 — Proaktive Skills & Event-Trigger

  Ziel: Timus beobachtet die Welt und handelt von selbst, ohne dass du was tust

  Was gebaut wird:
  - gateway/webhook_gateway.py — FastAPI Webhook-Endpoint
  - Event-Types: RSS_NEW_ITEM, PRICE_ALERT, EMAIL_RECEIVED, WEBHOOK_PUSH
  - Skill-Trigger: Event-Typ ↔ Skill-Mapping (konfigurierbar)
  - Beispiel: Neuer Artikel auf Heise.de → Research-Agent fasst zusammen → Telegram

  Abhängigkeit von M4: Self-Healing muss stabil sein, da proaktive Triggers keine User-Aufsicht haben

  Tests M5:
  ✓ pytest tests/test_webhook_gateway.py (neu)
  ✓ E2E: Webhook-Event → Skill ausgelöst → Ergebnis via Telegram?
  ✓ Kompatibilität: Webhook-Auth (HMAC) korrekt?
  ✓ Security-Test: Ungültiger Webhook → abgelehnt, kein Agent-Start?
  ✓ Integration: Proaktiver Task erscheint im Canvas-Store?
