Plan: Deep Research Engine — Produktionstauglichkeit (v6.0 → v7.0)

 Kontext

 Die Deep Research Engine liefert bei englischen technischen Themen (KI, ML, ArXiv) 0 verifizierte Fakten
 und irrelevante Inhalte. Ursache ist kein einzelner Bug, sondern ein Zusammenspiel von 5 strukturellen
 Problemen in der Verifikations-Pipeline. Das System ist ein "Schwergewicht" — 2700 Zeilen, 7 Quellen,
 3 LLM-Provider, DataForSEO-API — daher Milesteine mit Checkpoints.

 Lean 4 wird für formale Verifikation kritischer Algorithmus-Invarianten eingesetzt:
 - lean/CiSpecs.lean — CI-Theoreme (Std + omega, kein Mathlib, läuft in ~5s)
 - tools/lean_tool/tool.py — Mathlib-Specs (per Pre-Commit-Hook verifiziert)

 ---
 Architektur-Übersicht (Ist-Zustand → Soll-Zustand)

 Ist:  Query → [3 Web-Suchen (Location=DE)] → Relevanz (Score 0.4+) → DeepDive
            → Fakten (~5 pro Quelle) → Embedding-Gruppen (Threshold=0.85)
            → Verifikation (braucht 3+ Quellen) → 0 verified → leerer Report

 Soll: Query → [Language-Detect] → [5 Web-Suchen (Location=US wenn englisch)]
            → Relevanz (verbessert) → DeepDive
            → Fakten (~15 pro Quelle) → Embedding-Gruppen (Threshold=0.72)
            → Domain-aware Verifikation (moderate für Tech, strict für Fakten)
            → Corroborator als eigenständige Phase
            → ArXiv (Threshold=5 + Topic-aware Fallback)
            → Qualitäts-Gate: mind. 3 verified facts als Erfolg
            → Report mit echtem Inhalt

 -- Gewichtete Qualitäts-Score Invariante: 0 ≤ weighted_avg ≤ 1
 -- Gewichtung: authority=35%, bias=25%, transparency=15%, citations=15%, recency=10%
 theorem dr_weighted_score_in_bounds
     (a b c d e : ℝ)
     (ha : 0 ≤ a ∧ a ≤ 1) (hb : 0 ≤ b ∧ b ≤ 1) (hc : 0 ≤ c ∧ c ≤ 1)
     (hd : 0 ≤ d ∧ d ≤ 1) (he : 0 ≤ e ∧ e ≤ 1) :
     0 ≤ 0.35*a + 0.25*b + 0.15*c + 0.15*d + 0.10*e ∧
     0.35*a + 0.25*b + 0.15*c + 0.15*d + 0.10*e ≤ 1 := by
   constructor <;> nlinarith [ha.1, ha.2, hb.1, hb.2, hc.1, hc.2, hd.1, hd.2, he.1, he.2]

 Milestone 4: Verifikation & Corroborator (RC1 + RC3) — KERN-FIX

 Ziel: Domain-aware Verifikation + Corroborator vor dem Threshold-Filter.

 Modifizierte Datei: tools/deep_research/tool.py
 - _deep_verify_facts() (Z. 452–580): Domain-aware Modi
 - Corroborator-Aufruf (Z. 516–522): Aus Catch-22 befreien

 Was genau:
 1. Verifikations-Schwellen:
   - strict:   verified ≥ 3, tentative ≥ 2
   - moderate: verified ≥ 2, tentative ≥ 1  ← NEU
   - light:    verified ≥ 1, tentative ≥ 1
 2. Auto-Mode: Tech-Domain + strict-Input → effective_mode = "moderate"
 3. Corroborator für ALLE Fakten mit source_count ≥ 1 (nicht nur verified)
 4. Upgrade-Log: unverified → tentative wenn Corroborator Konsistenz bestätigt

 Neue Tests: tests/test_dr_verification.py
