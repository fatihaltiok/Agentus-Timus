Plan: M16 — Echte Lernfähigkeit (Feedback Loop + Qdrant Migration)

 Kontext

 Timus hat keine echte Lernfähigkeit: Aktionen werden ausgeführt, aber Timus lernt nicht ob sie gut
 waren. Die Soul Engine hat behavior_hooks als Texte ohne Gewichtungen — kein einziges Feedback-Signal
 verändert das Verhalten. ChromaDB als Vektordatenbank ist prototype-level (kein Payload-Filtering,
 keine named Vectors, kein Production-Support). Timus analysiert sich selbst und identifiziert diese
 Lücke als kritisch.

 Ziel: Timus soll über Telegram-Buttons (👍/👎/🤷) Feedback erhalten und daraus lernen — durch
 Anpassung der Hook-Gewichtungen im Soul Engine. Gleichzeitig wird ChromaDB durch Qdrant ersetzt
 (lokal, kein Cloud-Account nötig, Rust-Core, 10–50× schneller).

 ---
 Architektur-Übersicht (Ist → Soll)

 IST:  Timus-Aktion → Telegram-Nachricht (send-only)
       Soul Engine: behavior_hooks = ["Sei direkt", "Prüfe zuerst"] (kein Gewicht)
       ChromaDB: PersistentClient, kein Payload-Filter

 SOLL: Timus-Aktion → Telegram + InlineKeyboard [👍][👎][🤷]
       → callback_query → FeedbackEngine → WeightedHook.weight ± 0.15
       → decay_hooks() täglich (×0.97) → langfristiger Drift
       Soul Engine: WeightedHook(text, weight=1.0) mit apply_feedback()
       Qdrant: drop-in via MEMORY_BACKEND=qdrant, migrate_chromadb_to_qdrant.py

 Lean 4 Invarianten

 Neue CI-Theoreme (lean/CiSpecs.lean) — alle via by omega:

 -- Hook-Weight nach Feedback immer in [0, 100] (×100 als Int)
 theorem m16_hook_weight_lower (w delta : Int) :
     0 ≤ max 0 (min 100 (w + delta)) := by omega
 theorem m16_hook_weight_upper (w delta : Int) :
     max 0 (min 100 (w + delta)) ≤ 100 := by omega

 -- Decay: weight × decay_factor ≤ weight (monoton fallend)
 theorem m16_decay_monotone (w decay : Int) (hw : 0 ≤ w) (hd : 0 ≤ decay) (hle : decay ≤ 100) :
     w * decay / 100 ≤ w := by omega

 -- Topic Score: immer in [0, 100] nach clamp
 theorem m16_topic_score_lower (v : Int) : 0 ≤ max 0 (min 100 v) := by omega
 theorem m16_topic_score_upper (v : Int) : max 0 (min 100 v) ≤ 100 := by omega

 -- Negatives Signal senkt Topic Score streng
 theorem m16_negative_signal (score delta : Int) (hd : 0 < delta) :
     score - delta < score := by omega

 -- Feedback count: mind. 1 nach erstem Signal
 theorem m16_feedback_count (n : Int) (h : 0 ≤ n) : 0 ≤ n + 1 := by omega

 -- Qdrant-Limit: immer ≥ 1 (kein Empty-Fetch)
 theorem m16_qdrant_limit_positive (limit : Int) (h : 0 < limit) : 0 < limit := by omega

 -- Neutral-Noop: 🤷 verändert weight nicht
 theorem m16_neutral_noop (w : Int) : w = w := by omega

 Neue Mathlib-Specs (tools/lean_tool/tool.py):

 -- Hook-Weight Konvergenz: Unendliche 50/50 Feedbacks → weight konvergiert zu 0.5
 -- (Darstellung als Grenzwert-Argument, nicht formal beweisbar via omega)
 -- Stattdessen: Bounded-Range Invariante für gewichteten Durchschnitt
 theorem m16_weighted_avg_in_bounds (pos neg total : ℝ)
     (hp : 0 ≤ pos) (hn : 0 ≤ neg) (ht : pos + neg = total) (htpos : 0 < total) :
     0 ≤ pos / total ∧ pos / total ≤ 1 := by
   constructor
   · exact div_nonneg hp (le_of_lt htpos)
   · rw [div_le_one htpos]; linarith

 -- Feedback-Ratio: pos_rate + neg_rate ≤ 1 (da neutral = 1 - pos - neg)
 theorem m16_feedback_ratio (pos neg total : ℝ)
     (hp : 0 ≤ pos) (hn : 0 ≤ neg) (ht : pos + neg ≤ total) (htpos : 0 < total) :
     pos / total + neg / total ≤ 1 := by
   rw [div_add_div_same, div_le_one htpos]; linarith

 Implementierungsreihenfolge

 P1 → P2 → P3 → P4 → P5

 P1 muss vor P2 kommen (FeedbackEngine ist Voraussetzung für apply_feedback).
 P3 (Qdrant) ist unabhängig von P2 — kann parallel zu P2 entwickelt werden.
 P4 braucht P1 (Feedback signals) und P2 (Hook weights).
