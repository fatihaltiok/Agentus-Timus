  Was Lean 4 hier wirklich macht                            
                                                                                                                                                                                              
  Lean beweist nicht direkt den Python-Code. Es beweist mathematische Theoreme über die Invarianten die im Code gelten sollen.

  Python-Code  ──────────────────────────────────────────►  läuft
                                      ↑
  Du formalisierst die Invariante als Theorem
                                      ↓
  Lean 4  ─────────────────────────────────────────────►  beweist die Logik

  Beispiel konkret:

  # Python-Code in email_autonomy_engine.py:
  def evaluate(self, confidence, threshold):
      if confidence < threshold:
          return False  # nicht senden

  -- Lean-Theorem dazu:
  theorem m14_confidence_threshold (conf threshold : Int) (h : conf < threshold) :
      ¬ threshold ≤ conf := by omega

  Lean beweist: "Wenn conf < threshold gilt, dann kann threshold ≤ conf logisch nie gleichzeitig wahr sein." Das ist mathematisch unwiderlegbar.

  ---
  Was das bedeutet — und was nicht

  Was Lean garantiert:
  - Die Invariante ist in sich logisch schlüssig — kein Widerspruch möglich
  - Die Grenzfälle (Bounds, Thresholds, Retry-Limits) können strukturell nicht verletzt werden
  - Du hast formal nachgedacht was der Code garantieren muss

  Was Lean nicht macht:
  - Er liest deinen Python-Code nicht direkt
  - Er kann nicht prüfen ob dein Python die Invariante auch wirklich umsetzt
  - Die Brücke zwischen Theorem und Code bist du

  ---                                                                                                                                                                                         
  Weg 1 — Dafny (der pragmatischste)                        
                                                                                                                                                                                              
  Dafny ist Microsofts Verifikationssprache. Sie kann direkt Python-Code generieren aus verifiziertem Code:

  // Dafny: Spec + Implementierung zusammen
  method EvaluateConfidence(conf: int, threshold: int) returns (send: bool)
    ensures send <==> conf >= threshold  // formale Garantie
  {
    send := conf >= threshold;
  }

  dafny generate-tests EverythingIsVerified /target:py
  # → generiert verifizierten Python-Code

  Der Python-Code der rauskommt ist per Konstruktion korrekt — weil Dafny zuerst beweist, dann generiert. Das ist echter Brückenschlag.

  ---
  Weg 2 — Nagini (Python direkt verifizieren)

  Nagini (ETH Zürich) verifiziert Python-Code direkt mit Viper-Logik:

  from nagini_contracts.contracts import *

  def evaluate_confidence(conf: int, threshold: int) -> bool:
      Ensures(Result() == (conf >= threshold))  # formale Spec
      return conf >= threshold

  nagini email_autonomy_engine.py
  # Prüft: Implementierung erfüllt die Spec — mathematisch

  Das ist dem was du willst am nächsten — Python-Code direkt formal verifiziert.

  ---
  Weg 3 — Lean 4 Extraktion (der "richtige" Weg)

  Lean 4 kann Code extrahieren — allerdings nach Lean/C, nicht direkt nach Python. Der Workflow:

  Lean 4 Algorithmus schreiben
      ↓ beweisen
      ↓ lean --export
      ↓ C-Code generieren
      ↓ Python-Wrapper drumherum

  Für Timus bedeutet das: kritische Funktionen (z.B. confidence_check, whitelist_guard) in Lean implementieren, beweisen, als C-Extension einbinden. Aufwändig aber maximal korrekt.

  ---
  Weg 4 — Hypothesis (pragmatisch, sofort nutzbar)

  Hypothesis generiert tausende Testfälle aus deinen Lean-Invarianten automatisch:

  from hypothesis import given, strategies as st

  # Lean-Theorem: conf < threshold → kein Senden
  @given(conf=st.integers(), threshold=st.integers())
  def test_confidence_invariant(conf, threshold):
      engine = EmailAutonomyEngine()
      if conf < threshold:
          assert not engine.evaluate(conf, threshold).should_send

  Hypothesis sucht aktiv nach Gegenbeispielen — anders als normale Tests die nur bekannte Fälle prüfen. Das schließt die Lücke zwischen Lean-Theorem und Python-Code erheblich.
