Ich habe mich in den Ist-Zustand von timus eingearbeitet. Ich habe nichts geändert.

  Kurzbild
  timus ist im Kern ein Python-Multi-Agent-System mit FastAPI/MCP-Server als Zentrale, einem Dispatcher für Intent-Routing, spezialisierten Agenten auf einer gemeinsamen BaseAgent-
  Architektur, einem großen Tool-Layer, plus Autonomie-, Memory- und Canvas-UI-Schicht.

  Tatsächlicher Hauptfluss

  1. HTTP/UI/JSON-RPC kommt im MCP-Server an: server/mcp_server.py:602
  2. Der Server lädt Tools, registriert Agent-Spezifikationen lazy und exponiert Chat, SSE, Voice, Camera, Settings, Autonomy und JSON-RPC: server/mcp_server.py:820
  3. Für Chat-Anfragen ruft der Server den Dispatcher auf: server/mcp_server.py:1237
  4. Der Dispatcher entscheidet per Keyword-Fastpath oder LLM den Agenten und startet ihn: main_dispatcher.py:1091, main_dispatcher.py:1164
  5. Der Agent läuft in einem ReAct-ähnlichen Loop, ruft Tools via JSON-RPC auf und beendet mit Final Answer: agent/base_agent.py:1486
  6. Delegation zwischen Agenten läuft über die Registry, inkl. Loop-Schutz, Timeout, Blackboard-Write und Parallel-Fanout: agent/agent_registry.py:50, agent/agent_registry.py:770

  Basislogik der Agenten
  Die Kernlogik sitzt fast vollständig in agent/base_agent.py:

  - Capability-gesteuerte Tool-Sicht pro Agent

• 1. Agentenfluss Ende-zu-Ende

  Der produktive Hauptpfad ist:

  1. Eingang
     User-Input kommt über /chat oder Terminal rein und landet im Server: server/mcp_server.py:1237, timus_terminal.py
  2. Routing
     Der Server ruft get_agent_decision() auf. Das ist ein zweistufiger Router:
      - schneller Keyword-Fastpath
      - sonst LLM-basierte Agentenwahl
        Siehe main_dispatcher.py:1091
  3. Agent-Start
     run_agent() setzt Policy-Gates, Session/Lane, Canvas-Logging und instanziiert den Zielagenten oder einen Spezialpfad wie VisualNemotron: main_dispatcher.py:1164
  4. Agentenlauf
     Normale Agenten laufen über BaseAgent.run():
      - Blackboard-Kontext anhängen
      - Memory-Fastpath prüfen
      - evtl. strukturierte Navigation
      - LLM-ReAct-Loop
      - Action: {...} parsen
      - Tool per JSON-RPC aufrufen
      - Observation zurück an LLM
      - Abschluss per Final Answer: ...
        Siehe agent/base_agent.py:1486
  5. Tool-Aufruf
     _call_tool() validiert lokal, prüft Policy, sendet dann HTTP-JSON-RPC an den MCP-Server: agent/base_agent.py:594
  6. Server-seitiger Tool-Dispatch
     Der MCP-Server prüft Policy und Parametervalidierung erneut und dispatcht dann via async_dispatch in die globale Tool-Registry: server/mcp_server.py:1849
  7. Delegation
     Wenn ein Agent delegate_to_agent aufruft, landet das in der Registry:
      - Alias-Normalisierung
      - Delegation-Policy-Gate
      - Loop-Prevention
      - Timeout/Retry
      - Session-Weitergabe
      - Canvas/SSE-Events
      - Rückgabe als strukturiertes Dict
        Siehe tools/delegation_tool/tool.py, agent/agent_registry.py
  8. Parallel-Delegation
     Für unabhängige Subtasks gibt es delegate_multiple_agents, das auf agent_registry.delegate_parallel() geht: tools/delegation_tool/parallel_delegation_tool.py, agent/
     agent_registry.py:582

  Wo Kommunikation und Delegation heute konkret schwach sind

  1. Blackboard-Write im Delegationspfad ist faktisch kaputt
     In agent/agent_registry.py:483 ruft _auto_write_to_blackboard() bb.write(...) mit agent_id= auf und ohne topic=.
     Die echte Signatur erwartet aber agent, topic, key, value...: memory/agent_blackboard.py:86
     Ergebnis: Delegationsergebnisse landen wahrscheinlich nicht im Blackboard. Das ist ein echter Kommunikationsverlust zwischen Agenten.
  2. Der Delegationsvertrag ist nur halb strukturiert
     Die Registry liefert zwar status, agent, quality, metadata, blackboard_key, aber das eigentliche Arbeitsergebnis bleibt meistens ein Freitext in result: agent/agent_registry.py:424
     Danach wird metadata per Regex aus Text extrahiert: agent/agent_registry.py:502
     Das ist fragil. Für Orchestrierung ist das zu weich.
  3. Parallel-Fan-In ist gebaut, aber im Hauptpfad nicht sauber integriert
     ResultAggregator existiert: agent/result_aggregator.py
     Ich finde aber keinen produktiven Aufrufpfad dafür, nur Tests und Doku.
     Das heißt: Parallel-Ergebnisse werden zwar gebündelt, aber nicht systematisch in eine Meta-lesbare, kompakte Form überführt.
  4. Session-Kontext ist nicht durchgehend einheitlich
     Es gibt mehrere Session-Mechaniken:
      - conversation_session_id auf Agenten
      - session_id in Delegation/Research
      - Blackboard
      - Canvas-Session
        Das funktioniert teilweise, aber nicht als ein klarer trace_id/session_id/delegation_id-Vertrag.
  5. Es gibt mehrere MCP-Client-Pfade statt einer sauberen Transport-Schicht
     Es existiert ein vereinheitlichter Client in agent/shared/mcp_client.py, aber BaseAgent postet weiterhin selbst direkt via httpx, und andere Agenten haben teils eigene Pfade.
     Das erschwert Timeouts, Retries, Observability und Fehlernormalisierung.
  6. Agent-Namensraum ist leicht inkonsistent
     Dispatcher spricht häufig von development, Registry intern von developer: main_dispatcher.py:330, agent/agent_registry.py:61
     Das ist lösbar, aber für Delegation und Prompting unnötige Reibung.

  Was ich für die Optimierung priorisieren würde

  1. Delegationsergebnisse als hartes JSON-Schema statt Freitext plus Regex
     Jeder delegierte Agent sollte neben result verpflichtend artifacts, facts, next_actions, files, confidence, needs_followup liefern.
  2. Blackboard-Fix im Registry-Pfad
     Das ist ein direkter Bug und sollte zuerst weg.
  3. Fan-In wirklich anschließen
     delegate_multiple_agents() sollte seine Rückgabe direkt mit ResultAggregator oder einem ähnlichen kompakten Envelope für den Meta-Agenten aufbereiten.
  4. Einheitliche IDs pro Task
     session_id, trace_id, delegation_id, parent_task_id sollten durchgängig durch Dispatcher, Registry, Tools, Canvas und Blackboard laufen.
  5. Transport vereinheitlichen
     Alle Agenten sollten denselben MCP-Client und dieselben Retry-/Timeout-/Error-Regeln nutzen.

  2. Tool-System und MCP-Protokoll

  Praktisch ist timus kein klassisches externes MCP wie STDIO-Tooling, sondern ein eigener FastAPI-basierter JSON-RPC-Server. Intern heißt das zwar MCP, technisch ist es primär:

  HTTP + JSON-RPC 2.0 + ToolRegistryV2

  Tool-Lebenszyklus

  1. Tool definieren
     Ein Tool wird mit @tool(...) registriert und bekommt:
      - Name
      - Beschreibung
      - Parameter
      - Capabilities
      - Kategorie
      - optional Timeout/Priorität
        Siehe tools/tool_registry_v2.py
  2. Tool laden
     Der Server importiert beim Start alle Tool-Module aus TOOL_MODULES: server/mcp_server.py:145
  3. Dual-Registrierung
     Die Registry hängt Tools nicht nur intern ein, sondern bridged sie auch in jsonrpcserver.global_methods: tools/tool_registry_v2.py
     Das ist der Kern des Protokolls.
  4. Schema-Export
     Der Server kann Tool-Schemas für OpenAI und Anthropic ausgeben:
      - server/mcp_server.py:911
      - server/mcp_server.py:927
  5. Tool-Beschreibungen für Agenten
     Agenten bekommen nicht blind alle Tools, sondern eine textuelle Manifest-Beschreibung plus Capability-Filter: server/mcp_server.py:888, agent/base_agent.py:40

  MCP-/JSON-RPC-Aufruffluss

  1. Agent erzeugt Action: {"method": "...", "params": {...}}
  2. BaseAgent._call_tool() validiert und sendet POST auf /: agent/base_agent.py:594
  3. Server prüft Policy und Params: server/mcp_server.py:1849
  4. async_dispatch() ruft die registrierte Toolfunktion auf
  5. JSON-RPC-Resultat geht zurück an den Agenten
  6. Der Agent füttert die Observation wieder in den LLM-Loop ein

  Stärken des aktuellen Tool-Systems

  - saubere Metadaten-Registry
  - Capability-basierte Tool-Sicht
  - doppelte Validierung: Agent-seitig und Server-seitig
  - Schema-Export für verschiedene LLM-Provider
  - gute Grundlage für Policy-Gates und Observability

  Schwächen des aktuellen Protokolls

  1. Zu viel Text statt strukturierter Semantik zwischen Agent und Tool
  2. Kein einheitlicher Response-Envelope über alle Tools hinweg
  3. Mehrere Client-Implementierungen statt einer Transport-Schicht
  4. Parallel-Ergebnisse werden nicht als First-Class-Protokollobjekt weitergereicht
  5. Tool-Antworten sind für LLM lesbar, aber nicht immer für Orchestrierung stabil genug

  Saubere Zielrichtung
  Ich würde das Tool-/Delegationsprotokoll auf einen festen Envelope bringen, etwa:

  {
    "status": "success|partial|error",
    "summary": "...",
    "data": {},
    "artifacts": [],
    "metadata": {},
    "warnings": [],
    "next_actions": []
  }

  Dann:

  - Tools liefern diesen Envelope
  - Agenten liefern denselben Envelope bei Delegation
  - Meta orchestration arbeitet auf data/artifacts/next_actions, nicht auf Freitext
  - Regex-Metadatenextraktion kann fast komplett weg

  Fazit
  Für korrekte Agentenausführung liegt das Hauptproblem nicht im groben Architekturmodell, sondern in der Qualität des Kommunikationsvertrags zwischen:

  - Dispatcher
  - Agent
  - Tool
  - delegiertem Agent
  - Meta-Agent beim Fan-In

  Die kritischsten echten Punkte sind:

  1. kaputter Blackboard-Write im Delegationspfad
  2. zu weicher Delegationsvertrag
  3. fehlende produktive Fan-In-Aufbereitung
  4. uneinheitliche MCP-Transportpfade

  Wenn du willst, kann ich im nächsten Schritt direkt die technische Umsetzungsstrategie dafür ausarbeiten oder gleich mit Phase 1 anfangen:

  1. Blackboard-Bug fixen
  2. ein hartes AgentResult/Tool-Envelope-Schema einziehen
  3. delegate_multiple_agents() mit ResultAggregator produktiv verdrahten
